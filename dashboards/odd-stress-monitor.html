Dashboard · Methodology

How hostile is this road for an ADAS/AV System?

A living map of Operational Design Domain stress across San Francisco limits - built to quantify the conditions that push ADAS/AV systems toward their operational limits.

01 · The problem

ODD boundaries are invisible until it’s too late

Every ADAS system and autonomous vehicle operates within an Operational Design Domain - speed limits, weather, road markings, intersection type. When real-world conditions drift outside that envelope, the system either degrades silently or disengages.

The gap: engineers know their ODD on paper, but there is no live signal telling them how stressed any given zone is right now , how close it is to the edge of what the system can handle.

This dashboard fills that gap. It aggregates live incident data, work-zone geometry, weather observations, road-level congestion, and static intersection complexity into a single per-hexagon stress score - updated every hour, covering the full city grid.

02 · KPIs

What is measured and why

Each metric was chosen because it directly narrows or widens the margin between current conditions and a system’s validated ODD boundary.

0-10
ODD Stress Index

Composite score per hex cell. 0 = nominal conditions, 10 = operating at or beyond validated envelope.

0-10
Congestion

HERE real-time flow data. Averaged per hex across all road segments. 0 = free flow, 10 = standstill.

m²
Work Zone Burden

Active WZDx work zones weighted by lane impact and affected length within each hex.

n
Active Incidents

511 SF Bay events classified as incidents, accidents, and hazards - severity-weighted count per hex.

03 · Stress score model

A weighted sum, normalized to a human scale

Each sub-score is independently normalized to [0, 1] before weighting, so no single source can dominate purely because of its units. The weighted sum is then scaled to [0, 10].

Formula

stress = (
  0.25 × congestion +
  0.21 × work_zone +
  0.12 × incidents +
  0.12 × weather +
  0.25 × intersection +
  0.05 × planned_event
) × 10

Weights

Traffic congestion
25%
Intersection complexity
25%
Work zone burden
21%
Active incidents
12%
Weather severity
12%
Planned event
5%

Traffic congestion and intersection complexity are weighted equally as the two largest contributors - congestion is the most dynamic, spatially dense signal for ADAS operational margin, while intersection complexity captures the static structural difficulty that defines a hex’s baseline stress floor. Planned events carry the smallest weight, reflecting their relatively predictable and bounded impact compared to the other sources.

04 · Data sources

Where the numbers come from

HERE Traffic API
Real-time flow data queried by city bounding box. jamFactor per road segment is averaged to the H3 hex cell. Covers freeways and major urban roads.
hourly
511 SF Bay
Single /traffic/events endpoint. Response is split internally by event_type into incidents vs. planned events. No separate incidents endpoint exists.
hourly
Caltrans WZDx
Work zone feed for Bay Area District 4. Geometry midpoint determines hex attribution. Work zone burden weights by lane impact and affected road length.
hourly
NOAA / NWS
Hourly forecast from the MTR gridpoint. Condition text is mapped to a severity score - dense fog, heavy rain, and snow carry the highest weights. Applied uniformly across all hex cells (city-wide signal).
hourly
OpenStreetMap
Signal count and pedestrian crossing density per hex, queried once via Overpass API and stored as a static lookup file. Refreshed manually when the road network changes.
static

05 · Product decisions

Why it was built this way

01
H3 hexagons, not named corridors

Named corridors require manual curation and don’t scale to city coverage. H3 cells tile the city uniformly, enable heatmap rendering, and allow corridor-level analysis by grouping adjacent cells without any hardcoded list.

02
Static site generation, no live server

The pipeline runs hourly on GitHub Actions, writes pre-computed JSON to the frontend repo, and the dashboard reads static files. Zero hosting cost, zero server maintenance, and sub-millisecond page loads since there are no API calls at render time.

03
Jam factor, not raw speed

Raw speed varies by road type , 25 mph is free-flow on a residential street and severe congestion on a freeway. jamFactor is already normalized by HERE against each segment’s historical free-flow baseline, making it road-type-agnostic and directly averageable across a hex cell.

04
OSM intersection data baked in, not live

Querying Overpass on every hourly run is slow and adds a fragile dependency. Signal and crossing counts change only when the road network changes , so they are pre-computed once, stored as a lookup file in the repo, and loaded by the pipeline at runtime without any network call.

06 · Known limitations

What this monitor does not capture

Weather is city-wide

NWS gridpoint data applies a single weather condition to all hex cells. Microclimatic variation , fog in the valley vs. clear on the ridge , is not captured.

HERE coverage gaps

Traffic flow data is dense on arterials and freeways but sparse on residential streets. Hex cells with no segments return no congestion score and fall back to zero.

Hourly granularity only

The pipeline refreshes once per hour. Sudden incidents that open and close within the same hour will not appear in the stress score history.

No sensor or camera data

Pedestrian density, cyclist volume, and non-motorized road users are not measured. These are significant ODD stress factors for urban AV deployment that API sources do not cover.
